# מסמך פרויקט מעודכן
## מערכת לניתוח והפקת דו"חות אזורי תעשייה במערך ניטור איכות מי תהום

**גרסה:** עדכנית לפי מצב העבודה הנוכחי  
**סטטוס:** מסמך עבודה מרכזי לפרויקט  
**מטרת המסמך:** לרכז בתצורה אחת את ההיגיון המקצועי, המתודולוגיה, מבנה הנתונים, עקרונות הניתוח, מבנה התוצר, והלקחים מפיילוט רעננה — כפי שגובשו עד כה.  
**עקרון ארכיטקטוני:** כל יכולת אנליטית במערכת היא **plugin עצמאי** המממש חוזה (Protocol) קבוע. הוספת יכולת חדשה = הוספת תיקייה, אפס שינויים בליבה.

---

## 1. מטרת הפרויקט

מטרת הפרויקט היא לפתח מערכת עבודה סדורה לניתוח ולהפקת דו"חות אזוריים עבור אזורי התעשייה במערך ניטור איכות מי התהום.

המערכת צריכה לאפשר:
- ניתוח מצב עדכני לכל אזור תעשייה
- ניתוח מגמות רב־שנתי
- שילוב מידע היסטורי ותכנוני
- זיהוי והערכת מקורות זיהום אפשריים
- הצגה ויזואלית ברורה, מבוססת מפה
- הפקה שנתית של דו"ח מקצועי, שיעבור הגהה, הרחבה, דיוק ואישור מומחה
- הפיכת כל דו"ח שנתי מאושר לחלק מן הקונטקסט של דו"חות השנים הבאות

במילים אחרות, לא מדובר רק ב"דשבורד", אלא ב**מנוע דו"חות אזוריים רב־שנתי**, עם זיכרון מקצועי מצטבר.

## 1א. ארכיטקטורה טכנית — Plugin-First

### עיקרון Open-Closed
ליבת המערכת (`core/`) מגדירה **חוזים יציבים** (Protocols) ו**תיאום** (Pipeline Orchestrator). כל יכולת ניתוחית — מ-forensics ועד trend detection — היא **plugin** שנרשם אוטומטית ופועל תחת אותו חוזה.

### שש נקודות הרחבה

| Protocol | תיקייה | תפקיד |
|---|---|---|
| `DataSource` | `plugins/data_sources/` | שליפת נתוני גלם (רשות המים, אקסל, מפל"ס, תאגידי מים) |
| `ContaminantFamilyAnalyzer` | `plugins/forensics/` | ניתוח קבוצתי לפי משפחת מזהם |
| `ForensicsModule` | `plugins/forensics/` | טביעת אצבע כימית לכל קידוח |
| `TrendDetector` | `plugins/trend_detection/` | זיהוי מגמות בסדרת זמן |
| `SourceAttributor` | `plugins/source_attribution/` | דירוג מקורות חשודים עם ניסוח זהיר |
| `ReportSection` | `plugins/report_sections/` | רינדור קטע אחד בדו"ח HTML |

### חוזי נתונים (Data Contracts)
כל הזרימה בין מודולים עוברת דרך **dataclasses מוגדרים** (`core/contracts.py`):
- `BoreholeReading` — מדידה בודדת
- `FamilyReport` — אינדקס + מגמות + fingerprint לכל קבוצה
- `FingerprintResult` — מזהם מוביל, משניים, תוצרי פירוק, יחסים
- `TrendResult` — סוג מגמה, p-value, שיפוע, רווח ביטחון
- `Attribution` — מתקן + ציון + ניסוח זהיר + דרג

### Pipeline Orchestrator
מנוע תיאום שמגלה plugins רשומים ומפעיל אותם בשרשרת:
1. Ingest ← DataSource plugins
2. Group ← ContaminantGrouper (core)
3. Analyze per family ← FamilyAnalyzer plugins
4. Fingerprint per well ← ForensicsModule plugins
5. Detect trends ← TrendDetector plugins
6. Attribute sources ← SourceAttributor plugins
7. Render report ← ReportSection plugins
8. Store context ← ContextStore (knowledge layer)

### הוספת Plugin עתידי — דוגמה
כדי להוסיף מודול **PFAS Forensics** בעתיד:
```
plugins/forensics/pfas/
├── __init__.py
├── plugin.py       ← implements ForensicsModule, @register_plugin("forensics", name="pfas")
├── contaminants.yaml
├── tests/test_plugin.py
└── README.md
```
הפעלה דרך `config.py`:
```python
PIPELINE_CONFIG["emek_hefer"]["forensics"].append("pfas")
```
אפס שינויים בליבה, אפס שינויים ב-plugins קיימים.

לכל אזור תעשייה יוגדר סט קידוחים רלוונטי כאיחוד של שני מסלולים מחייבים:

## 5. קיבוץ מזהמים לקבוצות

הניתוח אינו צריך להתבצע כרשימת אנליטים שטוחה, אלא לפי **קבוצות מזהמים**.

### 5.1 קבוצות בסיס מוצעות
- ממסים מוכלרים / VOCs תעשייתיים
- דלקים ומרכיבי דלק
- מתכות ומתכות למחצה
- יונים עיקריים / אנאורגני / מליחות
- PFAS
- מזהמים מתעוררים נוספים
- סמני שפכים / פרמצבטיקה, לפי צורך

### 5.2 אינדקס זיהום 0–8 (סולם לוגריתמי)

לכל מדידה מחושב **יחס לתקן מי השתייה** (ratio = measured / standard).
היחס ממופה לאינדקס שלם בהתאמה לשפה הרגולטורית של רשות המים:

| אינדקס | יחס לתקן | משמעות |
|---|---|---|
| 0 | ≤ 0.25 | רקע — מתחת ל-¼ התקן |
| 1 | 0.25–0.5 | עקבות |
| 2 | 0.5–1.0 | מתקרב לתקן |
| 3 | 1.0–2.0 | חריגה קלה |
| 4 | 2.0–5.0 | חריגה משמעותית |
| 5 | 5.0–15 | זיהום חמור |
| 6 | 15–50 | זיהום חמור מאוד |
| 7 | 50–71.8 | זיהום קיצוני |
| 8 | > 71.8 | זיהום קריטי (>7,180% מהתקן) |

**אינדקס קבוצתי = max(אינדקסים של חברי הקבוצה).**

הסולם מיושם ב-plugin `core/` (אינו plugin עצמו — הוא שירות ליבה שכל plugin משתמש בו).

### 5.3 עקרון החישוב
לכל תוצאה:
1. משייכים פרמטר לקבוצת מזהמים
2. מחשבים יחס לערך ייחוס / תקן → ממפים לאינדקס 0–8
3. לכל קידוח ולכל תאריך מזהים **מקסימום קבוצתי**
4. במקביל שומרים את **כל חברי הקבוצה**

### 5.4 עקרון ההצגה
במבט הראשי:
- מציגים קבוצה דומיננטית
- מציגים עוצמה קבוצתית
- מציגים מגמה

במבט המשני:
- פותחים את חברי הקבוצה עצמם

### 5.5 חריגים למסגרת
לא כל פרמטר חריג ראוי להופיע בליבת הדו"ח האזורי.  
לדוגמה, במסגרת רעננה הוחלט **להוציא את הבורון מן המסגרת הראשית** של הניתוח, משום שאינו חלק מן המסלול המרכזי הרלוונטי לדו"ח זה.

## 6. ניתוח דינמיקה ומגמות

### 6.1 עקרון יסוד
הערכת מצב לא תתבסס רק על ערך אחרון, אלא על **דינמיקה רב־שנתית**.

### 6.2 מדדים מומלצים
לכל קידוח ולכל קבוצה:
- ערך מקסימלי היסטורי
- ערך אחרון
- חציון
- שכיחות גילוי
- משך פעילות
- שנת שיא
- מגמה

### 6.3 סוגי מגמות (7)
| סוג | הגדרה | שיטת זיהוי |
|---|---|---|
| **עלייה** | מגמה חיובית מובהקת | Mann-Kendall p<0.05, ≥3 דיגומים |
| **ירידה** | מגמה שלילית מובהקת | Mann-Kendall p<0.05 |
| **יציבות** | אין מגמה, פיזור נמוך | אין מובהקות + CoV נמוך |
| **תנודתיות** | אין מגמה, פיזור גבוה | אין מובהקות + CoV גבוה |
| **אירוע / פולס** | spike חד עם חזרה לבסיס | ערך > 3× חציון + חזרה תוך 1–2 דיגומים |
| **החמרה מחודשת** | עלייה לאחר מגמת ירידה | changepoint detection (ruptures) אחרי מגמת MK שלילית |
| **הופעה חדשה** | היסטורית מתחת לסף גילוי, כיום גילוי עקבי | ≥3 גילויים רצופים אחרי ≥3 אי-גילויים |

### 6.4 שיטות סטטיסטיות (TrendDetector plugins)
כל שיטת זיהוי מגמות מיושמת כ-**TrendDetector plugin** עצמאי:

| Plugin | ספרייה | תפקיד |
|---|---|---|
| `mann_kendall` | `pymannkendall` | מגמה מונוטונית + Sen's slope |
| `changepoint` | `ruptures` | נקודות שבירה (re-escalation, pulse) |
| `linear` | `scipy.stats.linregress` | fallback לרגרסיה לינארית |

ה-pipeline מריץ את **כל ה-TrendDetectors הרשומים** ומסווג ל-7 הסוגים לפי עץ החלטה ב-core.
כלומר: הוספת trend detector חדש (למשל LSTM) = הוספת plugin תחת `plugins/trend_detection/lstm/`, ללא שינוי בשיטות הקיימות.

### 6.5 פרשנות
לכל קידוח תופק פסקת פרשנות קצרה שתשלב:
- מצב נוכחי
- מגמה
- קונטקסט מדו"ח 2021
- משמעות אזורית
- האם הקידוח תומך בפלומה קיימת, מקור נוסף, או מסלול נפרד

## 8. טביעת אצבע של הזיהום

### 8.1 עקרון (ForensicsModule plugins)
לכל קידוח יש להפיק **טביעת אצבע כימית** באמצעות ה-ForensicsModule plugin הרלוונטי למשפחת המזהם הדומיננטית:
- מזהם מוביל
- מזהמי משנה
- תוצרי פירוק / חומרים נלווים
- מבנה יחסי בתוך הקבוצה
- שינוי בטביעת האצבע לאורך השנים

כל ForensicsModule מחזיר `FingerprintResult` (data contract אחיד), כך ש-forensics לממסים מוכלרים ו-forensics ל-PFAS מפיקים תוצר בפורמט זהה שה-pipeline מעבד באופן אחיד.

### 8.2 שימושים
טביעת האצבע תאפשר:
- להבחין בין פלומות שונות
- לזהות קידוחים המשתייכים כנראה לאותו מסלול
- לזהות קידוחים עם חתימה שונה המצביעה על מקור נוסף
- להבדיל בין מסלול תעשייתי כלורינטי, מסלול דלקי, מסלול PFAS וכדומה

### 8.3 קשרים בין קידוחים
הניתוח צריך לכלול ניסיון זהיר לענות על שאלות כגון:
- האם שני קידוחים מציגים חתימה דומה?
- האם מדובר באותו מסלול אך בעוצמה שונה?
- האם קידוח אחד משקף מסלול שונה?
- האם יש קידוח שמייצג ביטוי מרוחק יותר של אותה פלומה?
- האם קיימת תת־פלומה נבדלת?
- האם יש לפחות שתי מערכות זיהום נפרדות במרחב?

## 9. מקורות זיהום אפשריים

### 9.2 מדרג עדיפות

### 9.3 עקרון פרשני (SourceAttributor plugins)
הקשר בין אתר לבין זיהום צריך להיות מנוסח בזהירות.
ה-SourceAttributor plugin מחזיר `Attribution` עם שדה `cautious_phrase` שמוגבל לאוצר המילים הבא:
- **מועמד ליבה** — ציון כולל > 0.7
- **מועמד משני** — ציון 0.4–0.7
- **רקע מקומי** — ציון 0.2–0.4
- **לא תומך בפלומה המרכזית** — ציון < 0.2
- **מתאים למסלול נפרד** — חתימה כימית שונה מהפלומה הראשית

ניסוח זה **קריטי מבחינה משפטית-רגולטורית** בישראל — ייחוס מוקדם של מקור זיהום גורר השלכות.

## 11. מה הודגם בפיילוט רעננה

### 11.1 עקרונות שהוטמעו
בפיילוט רעננה הוטמעו העקרונות הבאים:
- נתונים עדכניים כבסיס הניתוח
- הטמעת 2021 כקונטקסט, לא כטאב נפרד
- הסרת בורון מן המסגרת הראשית
- הכנסת כיוון זרימה למפה ולפרשנות
- הוספת טביעת אצבע לכל קידוח
- הוספת קשרים אפשריים בין קידוחים
- הבחנה בין מסלול ממסים מוכלרים, מסלול דלקי ומסלול PFAS
- הצגה של כלל הקידוחים בקובץ
- שילוב Google Maps כשכבת בסיס

### 11.2 התמונה האזורית ברעננה
על פי המצב הנוכחי:
- המסלול האזורי המרכזי הוא **ממסים מוכלרים**
- `TCE` ו־`PCE` הם המזהמים המרכזיים בפלומה העיקרית
- `נד פז הנופר` מייצג מסלול **דלקי** משני ומקומי
- `נד תחנת טורבינות גז רעננה` מייצג מסלול **PFAS** חדש ונפרד
- `נת רעננה 1` נותר קידוח הליבה החמור ביותר
- `נת רעננה 3` מציג חתימה שונה יחסית והחמרה מחודשת
- `פ רעננה 25` עשוי לייצג ביטוי מרוחק יותר של המסלול הכלורינטי
- `פ רעננה 18` אינו חלק מלב המסלול התעשייתי

### 11.3 מסקנה אזורית מרעננה
ברעננה קיימות לפחות שלוש מערכות חתימה במרחב:
1. פלומת ממסים מוכלרים עיקרית
2. אות דלקי מקומי
3. אות PFAS נפרד

ובתוך הפלומה הכלורינטית עצמה ייתכן שקיימת שונות בין תתי־פלומה או יותר ממקור אחד.

## 14. נוסח מתודולוגי תמציתי

לכל אזור תעשייה ייבחר סט קידוחים רלוונטי כאיחוד של הקידוחים המופיעים בדו"ח המקורי ושל הקידוחים הנמצאים בתוך הפוליגון הגיאוגרפי של האזור. ניתוח המצב יתבסס תחילה על נתוני הניטור העדכניים ביותר, תוך קיבוץ הפרמטרים לקבוצות מזהמים, חישוב אינדקס זיהום 0–8 וניתוח מגמות רב־שנתי (Mann-Kendall + changepoint detection). דו"ח 2021 יוטמע בתוך הניתוח כקונטקסט פרשני רשמי לתמונת האזור בשנת 2021, ולא כלשונית השוואה נפרדת. דו"ח 2008 ישמש כהשלמה תכנונית והיסטורית. לכל קידוח תופק טביעת אצבע כימית באמצעות ForensicsModule plugin המתאים למשפחת המזהם הדומיננטית, ותיבחן מידת הדמיון או השונות בינו לבין קידוחים אחרים. כיוון הזרימה ישולב כפרמטר פרשני זהיר. במקביל יבוצע איתור שיטתי של מקורות זיהום אפשריים באמצעות SourceAttributor plugins, תוך דירוגם לפי 5 דרגי עדיפות וניסוח זהיר ("מועמד ליבה" / "מועמד משני"). התוצר יורכב מ-ReportSection plugins (דף אזורי, כרטיסי קידוח, נספח אנליטי, צ'ק-ליסט רגולטורי), וכל דו"ח שנתי מאושר ישמש שכבת קונטקסט רשמית לדו"חות השנים הבאות.

## 15. מיפוי מתודולוגיה → Plugin Architecture

| עיקרון מתודולוגי | Plugin / Core | הערה |
|---|---|---|
| קיבוץ מזהמים (סעיף 5) | `core/` — ContaminantGrouper | שירות ליבה, לא plugin |
| אינדקס 0–8 (סעיף 5.2) | `core/` — PollutionIndex | שירות ליבה |
| ניתוח מגמות (סעיף 6) | `plugins/trend_detection/` | כל שיטה = plugin נפרד |
| טביעת אצבע (סעיף 8) | `plugins/forensics/` | כל משפחת מזהם = plugin |
| זיהוי מקורות (סעיף 9) | `plugins/source_attribution/` | כל גישה = plugin |
| מבנה דו"ח (סעיף 10) | `plugins/report_sections/` | כל קטע = plugin |
| ידע מצטבר (סעיף 3.2) | `knowledge/` | שכבת ליבה, לא plugin |
| מקורות נתונים (סעיף 2) | `plugins/data_sources/` | כל מקור = plugin |
| כיוון זרימה (סעיף 7) | `core/` — FlowDirection | שירות ליבה, משמש plugins |
| קשרים בין קידוחים (סעיף 8.3) | `core/` — InterwellAnalyzer | שירות ליבה, משתמש ב-FingerprintResult |

**כלל אצבע:** מה שקבוע ואוניברסלי → `core/`. מה שמשתנה לפי משפחת מזהם, שיטת ניתוח, או מקור נתונים → `plugin`.
